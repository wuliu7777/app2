# 多平台实用工具集合 - 视频提取工具 设计方案

## 1. 概述
构建一个多平台（Windows, macOS, Android, iOS）的实用工具集合应用。首个功能为“全平台无水印视频提取”。采用 C/S 架构，自建开源后端服务解决视频解析难题，前端使用 Flutter 提供一致的跨平台体验。

## 2. 系统架构
*   **前端客户端**：Flutter (Dart)。负责跨平台 UI 展示、用户交互、调用系统播放器和文件下载系统。
*   **后端解析微服务**：Python/Node.js (基于开源项目如 Douyin_TikTok_Download_API 或 yt-dlp API)。提供统一的 RESTful API，接收原始链接，返回真实的无水印视频流地址、标题和封面。

## 3. UI 与交互流程设计
### 3.1 首页 (工具箱导航页)
*   **布局**：网格 (Grid) 布局，为未来添加新工具预留空间。
*   **内容**：目前展示一个“视频提取”的卡片/图标，点击进入视频提取页面。

### 3.2 视频提取页面
*   **输入区**：大尺寸的文本输入框，支持粘贴包含分享口令的混合文本。
*   **操作区**：醒目的“开始解析”按钮，附带 Loading 状态指示。
*   **结果区 (解析成功后展示)**：
    *   视频标题及封面图预览。
    *   “直接播放”按钮：调用内置播放器在线预览。
    *   “下载保存”按钮：将视频保存至本地相册或下载目录。

## 4. 核心数据流
1.  客户端正则提取用户输入中的 URL。
2.  客户端发起 HTTP GET/POST 请求至自建解析后端。
3.  后端处理反爬/签名/重定向，返回标准 JSON (含真实 video_url, cover_url, title)。
4.  客户端利用 video_url 进行流媒体播放或使用下载库进行二进制文件拉取。

## 5. 依赖库建议 (Flutter)
*   HTTP 请求：dio 或 http
*   视频播放：`video_player` (多端兼容)
*   文件下载与权限：dio (用于下载), path_provider, permission_handler, gal (保存至相册, 视具体需求选配)

## 6. 下一步计划
1. 选择并部署一个开源的后端解析服务。
2. 创建 Flutter 跨平台项目并完成 UI 框架搭建。
3. 联调后端 API，完成解析和下载流程。

## 7. 2026-07-02 联调修正记录
### 7.1 当前实际实现
*   后端入口为 `backend/main.py`，当前解析链路使用 `yt-dlp`，并通过 `/api/extract` 返回 `title`、`cover_url`、`video_url`、`source_url`。
*   前端入口为 `flutter_app/lib/api/video_api.dart`，固定请求 `http://127.0.0.1:8000/api/extract`。
*   Web 端下载通过 `http://127.0.0.1:8000/api/download` 代理触发，桌面端和移动端走本地保存流程。

### 7.2 已发现问题与修正
*   问题：真实平台分享文案通常是 `https://.../，打开 App`、`https://.../）。` 这类混合文本。旧后端正则 `https?://[^\s]+` 会把中文标点和后续说明文字一起当成 URL，导致 `yt-dlp` 收到错误链接并解析失败。
*   修正：`find_first_url()` 已改为在中文分享标点处截断，并清理尾部标点；同时给 `yt-dlp` 请求补充浏览器 `User-Agent` 与语言头。
*   回归测试：新增 `backend/test_main.py`，覆盖纯 URL、中文逗号分享文案、括号/句号包裹 URL 三种情况。

### 7.3 后续注意事项
*   修改后端后必须重启 `uvicorn`，否则前端仍会请求旧进程。
*   如果某个平台仍解析失败，先查看后端返回的 `detail`，再决定是否引入第二解析引擎；文档中曾提到的双引擎方案当前尚未完整落地。
*   后端依赖至少需要：`fastapi`、`uvicorn`、`yt-dlp`、`requests`。

## 8. 2026-07-02 B站优先支持方案
### 8.1 范围
*   当前先解决 B 站视频解析，支持 `www.bilibili.com/video/BV...` 与 `b23.tv` 短链。
*   后端会优先识别 B 站链接，不再把 B 站完全交给通用 `yt-dlp` 路径。

### 8.2 实现方式
1.  后端从输入中提取并清洗 URL。
2.  如果是 `b23.tv`，后端先跟随跳转拿到真实 B 站页面地址。
3.  后端从 URL 中提取 `BV` 号。
4.  调用 B 站 `x/web-interface/view` 获取 `cid`、标题、封面。
5.  调用 B 站 `x/player/playurl` 获取可播放地址。
6.  返回给前端的 `video_url` 使用本地 `/api/stream` 代理，代理层补充 `Referer` 与浏览器请求头，避免前端直连 B 站 CDN 被拒绝。

### 8.3 当前限制
*   当前版本优先使用 B 站返回的 `durl` 单文件视频流。
*   如果 B 站只返回 DASH 分离音视频流，当前版本会返回明确错误：暂不支持合流播放。
*   私密、付费、登录态限定、风控命中的视频不保证可解析。

### 8.4 回归测试
*   `backend/test_main.py` 已新增 B 站用例，验证 BV 号提取与 B 站 API 结果转成本地代理播放地址。
*   验证命令：`python -m unittest backend.test_main`。

## 9. 多平台支持路线
### 9.1 主流平台清单
*   已支持：B站、抖音。
*   第一梯队短视频/中视频：抖音、快手、小红书、西瓜视频、微博视频。
*   国际公开视频平台：YouTube、Vimeo。
*   长视频版权平台：腾讯视频、爱奇艺、优酷、芒果TV、搜狐视频。
*   社交/生态视频：微信视频号。

### 9.2 实现顺序
1.  B站：已接入专用 API 路径。
2.  抖音：已接入；短链跳转后走 `yt-dlp`，优先解决分享口令解析失败。
3.  快手：短链跳转后走 `yt-dlp`，失败时保留平台专属错误。
4.  小红书、西瓜视频、微博视频：先实现 URL 识别和短链规范化，再逐个平台验证解析器稳定性。
5.  YouTube、Vimeo：优先使用 `yt-dlp`，不处理登录/地区/年龄限制内容。
6.  腾讯视频、爱奇艺、优酷、芒果TV、搜狐视频：仅处理公开、非会员、非 DRM 内容；不做付费、登录态、DRM 绕过。
7.  微信视频号：需要单独评估，通常依赖登录态和客户端环境，默认不作为第一阶段目标。

### 9.3 统一规则
*   每个平台必须新增平台识别测试和解析结果测试。
*   解析失败必须返回明确平台名，不能只让前端显示“第 1 行解析失败”。
*   不支持绕过付费、登录、地区限制或 DRM。

### 9.4 抖音实现记录
*   支持域名：`www.douyin.com`、`v.douyin.com` 等 `douyin.com` 子域。
*   `v.douyin.com` 短链会先由后端跟随跳转。
*   抖音解析采用两段式：先尝试 `yt-dlp`；如果失败，再抓取抖音页面中的 `RENDER_DATA` / `__UNIVERSAL_DATA_FOR_REHYDRATION__` JSON，提取 `play_addr` 或 `download_addr`。
*   抖音 CDN 播放地址通过本地 `/api/stream` 代理，并补充 `Referer`。
*   后端返回 `platform: douyin`，便于前端或日志区分平台。
*   回归测试覆盖平台识别、短链跳转后调用解析器、页面 JSON 兜底解析。
*   前端已透传后端 `detail`，后续排查时可以直接看到平台失败原因。
